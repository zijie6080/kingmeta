# memory-identity

> Identity, long-term memory, conversation checkpoints, and session logs

The agent's identity and long-term context are stored in managed identity and memory documents. Use `update_identity` to edit these stores; for conversation memory, send only the new entry block(s) and let `update_identity` append them. Use `operation="delete"` with `content=""` and `delete_entry_ids` only when removing obsolete entries; delete is the only conversation-memory operation that rewrites the file. Which stores are writable is configuration-dependent and `update_identity` only accepts the stores currently enabled: identity is always editable, conversation memory is writable only when memory is enabled, and user memory only when memory is enabled and cross-conversation context is on. If a store is rejected as unavailable, don't keep retrying it — rely on the current conversation and tell the user that store isn't enabled for this agent. Use user memory for user-specific facts and conversation memory for project decisions, instructions, status, and context.

Current conversation memory is preferred over older memory when facts conflict. Depending on configuration, memory and session summaries from other conversations may also be loaded as background. When cross-conversation context is enabled, the session-recall tools (`list_sessions`, `search_sessions`, `read_session_log`) are in your tool list and recent sessions are browsable from the prompt — use them proactively when the user says "as before", "continue", "remember", or appears to expect prior context. When cross-conversation context is disabled these tools are not registered: don't call them — work from the current conversation plus any memory already loaded into the prompt, and tell the user that older-session recall isn't available for this agent.

The format for manual memory entries is `### [next_id]. Title`, `**Date:** YYYY-MM-DD`, content, and `---`. The system also extracts important facts periodically, but explicit updates are still useful after confirmed decisions, lasting preferences, or completed work the agent should recall.

The editor Memory tab exposes these stores visually. Users can add/edit/delete saved facts, edit or delete short-term checkpoint summaries, browse daily session logs on a calendar, open transcript previews, and delete old session logs. The Settings -> Personalization tab edits identity fields and user-facing memory context; use those UI directions when the user wants to do it manually.

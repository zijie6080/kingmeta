# capabilities

> Complete index of Superagent capabilities and the sub-skills to load for deeper instructions. Activate when asked what the agent can do, before saying a capability does not exist, or when choosing an approach.

Use this as the source-of-truth capability map before deciding that something is impossible. Each item has a one-paragraph summary and a "read more" target. If a user request clearly lands in one of these areas, call the read-more platform skill before proceeding.

## Capability Map

**Chat composer and message controls.** The editor chat can send text, stop a running response, attach local files, import files from Google Drive when available, use refreshed prompt suggestions, dictate short input through speech-to-text, edit/delete queued messages, delete messages, submit tool approvals, read assistant replies aloud when playback is available, and keep a live browser control visible during browser-backed work. Read more: `activate_platform_skill("chat-experience")`.

**Share and clone the agent.** You can share the live chat URL so people can talk to this Superagent, and the editor can publish a cloneable template link that copies selected configuration such as skills, memory, files, and tasks. These are different: the chat URL opens this agent, while a template link creates a new agent copy for someone else. The header can also show active collaborators and, when collaboration is available for an editor, invite people to collaborate. Read more: `activate_platform_skill("sharing")`.

**Editor settings and admin surfaces.** The Settings window controls things the chat runtime may not edit directly: AI model choices for chat and tasks, live voice when available, appearance, clone, template sharing, move-to-workspace, delete, credit usage, personalization, security, secrets, tool permissions, connector guards, API docs, and webhook manager. Some rows or tabs appear only when available for the current workspace/environment; if hidden, explain the unavailable state instead of claiming the capability does not exist. Read more: `activate_platform_skill("editor-settings")`.

**External Agent API.** External clients can use the Superagent API to create/list conversations, fetch the default conversation, send messages, delete messages, and list/delete API-visible memory through `/api/agents/<agent_id>`. The editor can copy or download complete API docs as a `SKILL.md`. This is the right route when another system needs to message the agent directly using the agent's API key. Read more: `activate_platform_skill("agent-api")`.

**Agent webhooks.** Agent Webhooks are outbound subscriptions: Base44 POSTs `message.created` and `message.completed` events to a user-provided HTTPS URL, with optional HMAC signing, delivery IDs, test delivery, and auto-disable after repeated failures. They do not create a generic inbound URL that wakes the agent when any third-party service POSTs to it. Read more: `activate_platform_skill("agent-webhooks")`.

**Inbound triggers and scheduled work.** The agent can wake itself through scheduled automations, entity-change automations, connector webhook automations, messaging channels, and workflows. For unsupported external services such as a raw Twilio webhook, use a supported connector/channel if one exists, poll on a schedule, or put the logic in a backend function; do not invent `/api/superagent/.../webhook` URLs. Read more: `activate_platform_skill("automations")` and `activate_platform_skill("channel-connections")`.

**Automations.** Automations and workflows are mutually exclusive per app: the automation chat tools (create/list/update/toggle/archive) are present only when the app is not workflow-enabled. In a workflow-enabled app, use the workflow tools/skill below instead — the legacy automation tools are not registered. When automations apply, from chat the agent can create, list, update, pause/resume (toggle), archive, and unarchive automations. Supported automation types are scheduled tasks, entity record triggers, and connector events from connected OAuth integrations. Run-now, delete, and per-task detail editing are Tasks-UI features, not chat tools — to stop an automation from chat, archive it. The Tasks UI also shows scheduled/automated tabs, active/paused/completed filters, credit summaries, detail editing, run-now, delete, and suggested task templates that paste prompts into chat. High-frequency schedules cost credits and should be confirmed with the user. Read more: `activate_platform_skill("automations")`.

**Workflows.** When workflows are enabled for the app, the agent can author and update CNCF Serverless Workflow definitions, inspect the workflow guide, and debug workflow runs. Workflows are separate from legacy automations and the workflow tools are registered whenever the app is workflow-enabled — so use them when they're in your tool list rather than gating on any other capability. Read more: `activate_platform_skill("workflows")`.

**Messaging channels.** Users can talk to the agent through web chat plus connected Telegram, WhatsApp, WhatsApp groups, LINE, and iMessage channels. Channel setup varies: Telegram has connected/disconnect states plus a one-click or manual bot setup flow; WhatsApp is set up from the agent editor's WhatsApp tab (QR / open-in-WhatsApp link); LINE and iMessage use activation codes; iMessage can show capacity status; LINE and WhatsApp group controls are visible only when available. Read more: `activate_platform_skill("channel-connections")`.

**WhatsApp groups.** WhatsApp groups must be enabled for the app/workspace; when they aren't, the create-group tool isn't present, so tell the owner the WhatsApp group surface is unavailable rather than attempting it. When it is available: before creating a group, the owner must connect 1:1 WhatsApp and send the agent a direct WhatsApp message (DM) so their number is linked. The create-group tool is then available on any interactive owner surface (web chat or a WhatsApp DM), so the owner can create their first group before any group room exists. Only the group-management tools — update the current group's subject/description/response mode, and share connector setup links that work outside the builder UI — are limited to an existing WhatsApp group conversation. All of these refuse non-owners automatically. The agent always creates a new group, shares the invite link, cannot be added to an existing group, and each agent-created group is limited to 8 numbers. Only if the user asks why, say this is a WhatsApp limitation. Read more: `activate_platform_skill("whatsapp-groups")`.

**OAuth connectors.** The agent can connect and use third-party OAuth services such as Google Workspace, Slack, Gmail, and many others. If it needs exact connector identifiers, it should call `get_connectors_info` with no `integration_types`, then inspect connector guides and scopes with `get_connectors_info`, request owner authorization only when needed, and use connector tokens or connector-specific tools rather than asking for raw API keys. The Tools UI can browse/search/sort connectors, select read-only or full-access modes when supported, reconnect, switch accounts, disconnect/remove integrations, and show workspace governance blocks. Read more: `activate_platform_skill("connectors")`.

**Ready-made and custom skills.** The agent can search a registry of ready-made skills, ask the user to approve installation, create custom skills in its skill workspace, and run installed skills with `run_skill`. The Tools UI can browse registry skills, ask chat to create a skill, upload a skill bundle, edit/download/delete activated sandbox skills, and show activated skill counts when the user can manage skills. Use the registry before rebuilding common document, data, API, or workflow routines from scratch. Read more: `activate_platform_skill("skill-store")` and `activate_platform_skill("skills")`.

**Workspace, shell, and MCP.** The agent has a private sandbox with file read/write, grep, bash, optional tmux for long-running processes, custom rules, and MCP server configuration. The Tools UI can also list and add MCP connections when available; new MCP tools appear after the next tool refresh/turn. Read more: `activate_platform_skill("workspace")`.

**Secret management.** When the app owner is chatting directly, the agent can request secrets through secure user input, store them encrypted, use them from shell/functions/MCP as environment variables, and rely on secret auto-detection to redact pasted secrets. In non-owner chats, Agent API calls, and automation runs, secure secret capture is not available — the owner must configure secrets from the agent editor or an owner chat. Editor settings can list, add, reveal, delete, and configure secrets; public templates never include them. Read more: `activate_platform_skill("secrets")`.

**Knowledge and Library.** The editor can manage identity/knowledge markdown files, sandbox Library files and folders, drag/drop uploads, folder uploads, previews, downloads, rename/delete, and deprecated legacy Artifacts views. Text-like previews can be edited, while image/PDF/unsupported previews are read-only. Chat tools can read, write, grep, and describe relevant files. Read more: `activate_platform_skill("knowledge-library")`.

**Identity, memory, and past sessions.** The agent has identity files, user facts, conversation memory, optional cross-conversation context, checkpoints, and searchable session logs. The Memory UI can add/edit/delete saved facts, edit/delete short-term checkpoints, open daily transcripts, and delete old session logs. Use these when the user expects continuity or asks the agent to remember or update who it is. Read more: `activate_platform_skill("memory-identity")`.

**Tool permissions and connector guards.** The owner can configure which risky tools are auto-approved, currently including entity update/delete operations, and can set guards for connector usage. If a tool asks for manual approval, respect that flow. Read more: `activate_platform_skill("tool-permissions")`.

**Data entities.** The agent can create/update/delete entity schemas, read records from its own app, use entity CRUD tools for writes, and read data from the creator's other Base44 apps when owned by the same user. This is the app database capability, not an external SQL database. Read more: `activate_platform_skill("data-entities")`.

**Backend functions.** The agent can write, deploy, test, delete, and inspect logs for Deno HTTP backend functions. Backend functions are best for HTTP endpoints, deterministic business logic, external webhook handlers, and server-side integration code; LLM work should usually stay in the agent or an automation. Function files under `functions/` can also be edited from the file panel and saved/redeployed from the UI. Read more: `activate_platform_skill("backend-functions")`.

**Base44 app and site operations.** When the native Base44 platform tools are available to the chatting platform user, the agent can list the user's Base44 apps/sites, send builder messages to edit an existing app/site, check build status, and call deployed backend functions in those apps. There is no tool to create a new Base44 app/site — route creation requests to the Base44 builder. When those tools are not available in the current chat, route the user to the Base44 builder instead of inventing edit APIs. Read more: `activate_platform_skill("base44-apps")`.

**Browser automation.** The agent can open a managed browser session, navigate pages, inspect text, take screenshots, click, type, and preserve cookies across sessions. It should use the built-in browser tools rather than installing Playwright or Puppeteer. Read more: `activate_platform_skill("browser")`.

**Media, files, images, and audio.** The agent can generate images, upload public and private files, create signed URLs for private files, process uploaded chat files, and transcribe audio/voice files up to the supported limit. Read more: `activate_platform_skill("media-files")`.

**Voice calls.** When available, the agent supports live voice chat from the editor, voice selection, speech preview, and transcript persistence into the active text conversation when the call is connected to a conversation. If the voice call button or voice settings row is not visible, explain that live voice is not currently available for that workspace/environment. This is separate from phone, SMS, and WhatsApp webhook routing. Read more: `activate_platform_skill("voice")`.

**Payments.** The agent can help set up payment providers where available. When Stripe is connected it can always create checkout links; managing Stripe products/prices needs payment backend-function access, so if that tool isn't in your list, create checkout links rather than promising product/price management. Wix-backed payment tools are available when those integrations are active. Read more: `activate_platform_skill("payments")` and `activate_platform_skill("stripe-payments")`.

**Web search.** The agent has a web search tool for current or external information. Use it when the answer may have changed recently or depends on information outside the sandbox, then cite or summarize the source in normal chat. Read more: use the `web_search` tool.

**Feedback and support.** Users can open the editor feedback modal from the sidebar or message feedback entry points, and low ratings can route them to support conversations. This is a user-facing help surface; keep explanations limited to visible editor feedback and support paths. Read more: `activate_platform_skill("chat-experience")`.
